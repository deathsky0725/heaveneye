## 2. "Resets in Xh" — where it comes from

**Source:** `UsagePanel.tsx:112` → `formatDuration(row.expiresIn ?? 0)`

The `expiresIn` value is computed in `UsagePanel.tsx:93`:
```ts
expiresIn: agg.expiresAt != null ? agg.expiresAt - now : null
```

`agg.expiresAt` comes from `state.getUsage5h()` → `nextResetAt` = `windowStartedAt + WINDOW_MS` (5h).

`windowStartedAt` = timestamp of the **oldest event still inside the 5h rolling window** (engine.ts:135).

**Reset countdown logic:**
- 5h rolling window — resets automatically after 5h from oldest event
- No external API involved — purely in-memory rolling window
- When no events exist, `expiresIn` is null → no countdown shown

## 3. Per-agent token attribution — what data exists

**FULLY AVAILABLE.** Three data layers:

### Layer A — In-memory rolling windows (engine.ts:51-57)
```ts
tokenEvents: Map<AgentId, TokenEvent[]>   // 5h window
tokenEvents24h: Map<AgentId, TokenEvent[]> // 24h window
```
`TokenEvent` shape: `{ ts: number, model: string, usage: TokenUsage }`

### Layer B — 5h window API: `GET /api/usage/5h`
Returns per-agent per-model aggregated usage (input/output/cacheRead/cacheCreate + windowStartedAt + nextResetAt).

### Layer C — Historical daily files (engine.ts:313-330)
`~/.heaveneye/usage/<agent>-<date>.json` — one file per agent per day.
Loaded by `_loadHistoricalDays()` → `_getUsageNdays()` → `getUsage7d()`.
Used for weekly/monthly aggregation (today uses in-memory 24h events).

## 4. Hardcoded cap constants — status

**Previous Verdict B concern:** No MiniMax quota API, caps hardcoded.

**Current status:**
- `CAP_5H = 2_500_000` (server/index.ts:63) — MiniMax "500k tokens/h × 5h" conservative estimate
- `CAP_WEEKLY = 10_000_000` (server/index.ts:87) — MiniMax plan weekly cap

These are hardcoded but reasonable estimates based on known MiniMax plan info. No live quota API exists from MiniMax.

**Impact on G1 backend:** The backend is already shipped and functional. The caps are static but correct for the known plan. This is not a blocker for G1 — it's a known limitation that would require a MiniMax quota API to resolve dynamically.
